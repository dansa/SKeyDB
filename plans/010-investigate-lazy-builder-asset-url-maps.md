## Why This Matters

Builder V2 uses asset helpers that eagerly create URL maps for card, portrait,
wheel, and wheel-mini image families. The browser still lazy-loads individual
images, so this is not an urgent image-download bug. It is a route-module and
bundle-shape investigation: opening Builder V2 pulls broad generated asset URL
maps even when only a few slots or one picker tab need them.

## Steps

### Step 1: Measure build impact first

Run a production build and record the relevant asset/module output. If the
current build output is too opaque, use Vite's existing output only; do not add
bundle analyzer dependencies in this plan.

**Verify**:
`npm run build:quiet` exits 0.

### Step 2: Choose the smallest safe strategy

Pick one strategy:

1. Generated static manifest: replace eager glob maps with a checked-in mapping
   generated by an existing or new script.
2. Lazy glob resolver: use non-eager `import.meta.glob` and return URLs through
   a small async/cache boundary only where the UI can tolerate it.
3. Split eager maps by asset family and import only where needed.

Default to **generated static manifest** or **split eager maps**. Avoid making
Builder V2 slot projections async unless the operator explicitly accepts that
UI complexity.

**Verify**:
The chosen strategy preserves synchronous `getAwakenerCardAsset`,
`getAwakenerPortraitAsset`, `getWheelAssetById`, and `getWheelMiniAssetById`
unless you have explicit approval to change callers.

### Step 3: Add focused asset-resolution tests

Add tests for:

- Known awakener card and portrait asset lookup.
- Known wheel full and mini asset lookup.
- Unknown asset fallback remains `undefined`.
- Explicit slug exceptions such as `24` still resolve.

Use existing domain test style. If no nearby asset test exists, create
`src/domain/awakener-assets.test.ts` and `src/domain/wheel-assets.test.ts`.

**Verify**:
`npx vitest run src/domain/awakener-assets.test.ts src/domain/wheel-assets.test.ts --run`
passes, adjusting paths to actual files.

### Step 4: Implement the selected strategy

Make the smallest change that reduces eager route-map coupling while preserving
helper APIs. Keep fallbacks and slug normalization unchanged.

If the build output does not improve or the implementation becomes more complex
than the current eager maps, stop and report that the current approach is not
worth changing yet.

**Verify**:
`npm run build:quiet` passes before and after; record any relevant output

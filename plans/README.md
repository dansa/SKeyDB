## Findings Considered And Rejected

- Core Builder V2 DnD/model correctness bug: rejected. A read-only correctness
  pass over the primary DnD/model paths found no current bug, and the focused
  Builder V2 suite passed.
- Builder V2 default-route opt-in test gap: rejected. `src/App.test.tsx` already
  covers opting into Builder V2 as default, redirecting `/builder` to
  `/builder-v2`, and preserving `/builder?classic=1`.
- Production dependency vulnerability: rejected. `npm audit --omit=dev --json`
  reported zero production vulnerabilities at generation time.
- Broad generic `BuilderV2AwakenerPicker` tile abstraction: rejected/covered.
  `BuilderV2PickerPrimitives.tsx` already centralizes the shared frame, button,
  art, caption, chip, and sort/toggle pieces; the per-kind labels, payloads, and
  chips are still clearer when explicit.
- Moving mobile picker dialog state into `useBuilderV2Model`: rejected. The model
  owns domain selection state, while the mobile layout owns dialog visibility,
  focus return, and viewport-specific retargeting.
- Separate focus-state consolidation: rejected. Builder V2 focus selectors and
  tokens are already centralized enough for the current drift level.
- Picker state-chip rewrite: rejected. State-chip tone variables are already
  centralized; the remaining maintainability risk is selected/emphasis drift,
  covered by 014.
- Broad generic card/tile surface recipe plan: merged into 014 and 015 only where
  selected state and density drift create real future churn. A wider visual
  surface abstraction would risk becoming fake structure before the next design
  pass.
- Immediate Builder V2 `KeyboardSensor` rollout: deferred. Official DnDKit docs
  cover keyboard sensors and announcements, but current Builder V2 policy is
  pointer DnD as progressive enhancement, click-first alternatives for non-pointer
  use, and no DnDKit keyboard-draggable advertisement. Keyboard DnD parity should
  be its own product/a11y plan if required.
- Mobile pointer DnD enablement: deferred. `BuilderV2DndCapability` deliberately
  disables pointer DnD on coarse/hoverless devices, and mobile picker/lineup
  flows are covered by click/tap plans 012 and 013.
- Rewriting the Builder V2 DnD resolver boundary: rejected. The pure
  `builder-v2-dnd.ts` resolver is already the maintainable rule boundary; the
  current risk is DnDKit candidate arbitration and feedback drift around it.
- Separate drag-overlay clipping plan: merged into 018. Overlay visibility and
  scroll/viewport clipping should be proven in the browser smoke while adding
  DnD-focused checks.
- Classic Builder DnD adjustment for posse drops: rejected/deferred after product
  clarification. Builder V1 is due for retirement, so the posse target issue
  should be fixed only if it becomes breaking before V2 catches up.
